Problem:

You took up salsa dancing because you’ve lost interest in coding, for now, and wanted to do something joyful and embodied.

But apparently coding hasn’t lost interest in you.

As it turns out, the dance floor is running a cursed implementation of Python where the interpreter executes each line the moment you type it. No debugging. No ctrl+z. No stack traces that you can read in private. Just pure, merciless real-time execution with immediate visible consequences, and then your brand-new line of code vanishes forever.

Worse: the interpreter is another human being, and she remembers.

The syntax is unforgiving. Every command must be issued with precise timing— not just correct, but correct NOW. Too early and the interpreter throws an exception. Too late and you've created a memory leak that cascades through the next several lines until you can write yet more code to correct it. The garbage collector is judging you.

You are writing code at 120 beats per minute. Each beat demands a new line. Skip a line? TimeoutError. Wrong indentation? The interpreter stumbles, recovers, but she gives you a look. The impulse to apologize is strong but must be resisted since every second you spend apologizing is a second you’re not spending on the next line of code, which is due to execute… now. Again. Again. Again. Until the music ends.

Memory allocation happens through physical contact. You need 8 bytes of space to invoke the InsideTurn() method but your interpreter only has 6 bytes available. You try to call the method anyway and accidentally cause your interpreter to shoulder-check another programmer, who gives you a dirty look. Oops.

There's no documentation. The API changes depending on who's interpreting. Some interpreters are running Python 2.7 and expect completely different syntax. Others have custom modifications you can only discover through careful runtime experimentation. One has a busted leftTurn() implementation that finishes late every time. Another implements all toss() commands as turn() commands. The other night a girl stopped you midway through writing a line of Python and, with an air of mixed pity and condescension, tried to correct its syntax into Javascript whereupon you terminated the program.

You're supposed to figure her situation out, whatever it is, within the first three lines of code and adjust accordingly.

Frequently there's an AttributeError and you're not quite sure if it's a problem with your code or with your interpreter. No time to think about that now, though, you have more lines of code to write. Maybe just avoid that syntax going forward?

You take note of other programmers around you on the dance floor. One of them is clearly just copypasting pass over and over, and neither him nor his interpreter seem to mind. Another keeps calling a really cool function that you didn’t know existed, which you resolve to reverse-engineer later. Another is writing line after line of incomprehensible code calling functions with names like tp_final_1() and ryuid_fast() and it’s somehow working better than anything you’ve seen before.

How the hell does anyone do this?

Solution:

Stop trying to write beautiful code. Write functioning code. The interpreter doesn't care about your clever list comprehensions - she cares that commands arrive on time, every time, with appropriate syntax, even if those commands are just pass. Especially if they're just pass.

Your saving grace is this: the interpreter WANTS to execute successfully. She’s not a hostile compiler hunting for errors. She’s desperately trying to make your garbage syntax work, adding implicit type conversions, filling in missing arguments, gracefully handling your exceptions. Every interpreter you've ever crashed was trying to help you succeed.

After a few hundred hours of stressed-out jamming at the keyboard and debugging your numerous failed commands with experts, you start to develop a sense for which functions are probably available - every interpreter has move(), most have turn() and crossbodyLead(), some have advanced libraries you can probe for with careful feature detection. Throw out a tentative function call with loose typing, watch how it executes, and you can immediately know whether she has the relevant module installed or if you need to ditch that whole set of functions entirely.

The real fun begins when you realize the better interpreters have absurdly permissive duck typing. You can, once you get the knack, stop calling specific functions and start calling the shape of functions. Not exactly a CrossBody instance but something that the interpreter can pretend is a CrossBody instance, and she figures out the implementation details.

The lovely part is when it works. You call a function you've never seen before, with parameters you're guessing at, and it executes beautifully. Not often. But sometimes.

You were never destined to escape programming.
