Preface on specific model usage

The following assumes you’ll be using Claude-3.5-Sonnet for all nontrivial operations except where specified. (I’ve found that for complex multipart tasks the O1 series does better work more reliably.)

Language and Framework Selection

  • Popularity of Language and Framework is important. You want languages and frameworks that the LLM has seen a lot of. Javascript, java, typescript, C#, Python— these are your friends. The more popular it is, the fewer errors you’ll make.

  • You want static typing if you’re interacting with a codebase of significant size; when Cursor makes sweeping changes it frequently either hallucinates methods that don’t exist or just misunderstands the types of things. If you have a compiler to catch these errors, the rest of the underlying logic is typically (with some exceptions) quite solid and you can just point cursor at the compilation errors and say “please fix.”

    • if you have a language with optional type-checking (python, for instance) this is almost as good since LLMs are excellent at annotating things with correct type information.

    • If you’re using a dynamic language without even optional type-checking you can still write small things and boilerplate, but this probably does make the dream of “LLMs write all my code for me” impractical as of the 10/17/2024 state of the art. Sorry.

Shit Cursor Is Bad At

  • Cross-class interactions. It just ignores spooky-action-at-a-distance interactions between mutable state unless explicitly prompted to consider them.

  • Telling what methods exist. From what I can tell, it really likes to hallucinate methods that it feels ought to exist, and it mostly just reads the method name and not the associated docstrings. That means you really need the method names to be clear as to what is being done!

  • It’s bad at noticing and proactively dealing with duplicated code and conceptual issues in code organization.

    • It doesn’t really like to refactor unless explicitly prompted; which means that if you’re doing the same thing N times it’ll look at those instances and be like “oh, cool, we just need to do that N+1 times”. It needs to be told when to remove duplication.

  • It’s bad at navigating long chains of method/property calls. Extremely frequent site of hallucinations.

    • If you routinely need to do shit like GameState.getInstance().combatState().combatResources().iron, then you should probably add a helper method to get that more directly since LLMs seem to have trouble with this kind of call pattern. Or you can just get used to frequently correcting it.

  • It currently can’t identify and pull relevant classes into the context on its own; it NEEDS you to identify these in advance or else it will just make stuff up.

  • It’s bad at remembering long parameter lists. I’ve started using the builder pattern because Claude has such an easier time with it.

Things Cursor Is Amazing At

Half-Assed 2-Second Code Reviews

“Is this code right?” is a question I ask Claude-3.5-Sonnet any time i’ve written anything remotely complicated, and very frequently it’ll tell me about bugs. (Sometimes it will hallucinate bugs or nitpick, which is fine.)

Sweeping Refactorings

This is a broad category but it includes things like:

“Hey, Cursor, I’ve decided that frequently-used LibraryMethod.xyz() is a janky piece of garbage and would like to work around it in this other way. Please execute this operation on the following classes.” You then add a prohibition against LibraryMethod.xyz() into your .cursorrules file.

You can also say:

”Please consolidate the bounding box calculation logic into a new class, then make these classes use it.”

You can even say:

“Make this UI prettier.” (This one is complex enough to want o1-preview, but it can do it.)

Consequently I make many, many more refactorings than I used to pre-Cursor.

Refactor Suggestions

“Hey, am I doing anything non-idiomatic in this class?”

or

"Is there a cleaner way of encapsulating my action handling logic?”

Logic Extraction From One Class Into >1 Class (o1 models preferred)

“This class is doing way too many things. Please take the logic here and pull it out into some number of other classes.” I literally did this with o1-preview.

Straightforward method implementation

“this method calls S3 and shoves an object in our storage account. write it for me.”

Log Statement Adder

“Please sprinkle this code liberally with detailed logs”.

Debugging (o1 preferred)

Copypaste the error and provide links to EVERY CLASS YOU SUSPECT IS INVOLVED.

UI Code, Surprisingly

Hey, I didn’t expect it either, but it gets UI code approximately correct most of the time! It won’t get things exactly to where you want them but it does get something basically-functional most of the time that I, the human, can then tweak.

The Kinds of Errors Cursor Makes

(And Their Severity, Based On My Own Hobby Game Project)

Most Frequent: Hallucinating method names and properties and messing up import paths

This is trivially fixed in statically-typed languages, since nearly always there’s a real method that is extremely similar to what cursor has hallucinated and you just need to swap out the fake method for the real one. Even so this happens quite a lot and is somewhat annoying.

Moderate Frequency: messing up details of subtle library call patterns, resulting in runtime errors

I’ve had to forbid Cursor from doing any of my event-listener-management code (via .cursorrules) because the rules in Phaser (the framework i’m using) for coding up listeners are very strict and disobeying them results in hard-to-debug stack traces bubbling up from deep within the framework code. Claude gets it wrong… sometimes, which is enough for pain.

Least Frequent: Actual Logic Errors That Can Be Discovered Through Just Reading Through The Implementation

This happens very rarely, in my experience; when it does happen it’s typically sufficient to point o1-mini at a failing unit test and the class it’s referring to and saying “please fix.“ I’ve gotten so lazy.
